# 26. Build Roadmap — исполняемые итерации

## Назначение

Этот файл превращает всю документацию (00–25) в последовательность
исполняемых итераций. Формат рассчитан на работу с AI-агентом: ты говоришь
**«сделай итерацию N»**, агент берёт её scope, строит код, сверяет результат с
Definition of Done и тест-кейсами из `23`, и коммитит.

Роадмап опирается на `12_release_plan.md` (продуктовый порядок) и
`22_development_checklist.md` (майлстоуны), но детализирован до уровня задач и
покрытия документов.

## Где живёт код (важно)

- Этот репозиторий (`playup-docs`) остаётся **только документацией** — код сюда
  не коммитим (см. `AGENTS.md`).
- Код приложений живёт в отдельных репозиториях (решение из `00`/`02`):
  - `playup-admin-app` — админка + общий backend/API (backend внутри admin-репо,
    как рекомендует `24`);
  - `playup-client-app` — Telegram mini app.
- По умолчанию агент создаёт их как соседние папки рядом с этим репозиторием и
  инициализирует git в каждой. Это решается в Итерации 0.

## Рекомендуемый стек (финализируется в Итерации 0)

- БД: PostgreSQL (`02`, `14`).
- Backend + Admin: Next.js (API routes + админ-UI в одном репо) — самый быстрый
  путь к рабочей петле; ORM Prisma. Альтернативы (NestJS/FastAPI) допустимы.
- Client: Vite + React как Telegram mini app (`03`, `22`).
- Дизайн-токены: из `25` (общий пакет/файл токенов, переиспользуется обоими UI).
- Это рекомендация. Если ты выберешь другой стек, скажи в Итерации 0 — роадмап от
  стека не зависит, меняются только инструменты.

## Как агент исполняет итерацию

1. Перечитывает документы из колонки «Покрывает».
2. Делает backend → данные → UI; не смешивает Booking/Attendance/Payment (`17`,`19`).
3. Прогоняет соответствующие manual-тесты из `23` и data-quality из `22`.
4. Коммитит в репозитории кода (не в docs), сообщение с префиксом
   `feat:`/`chore:` и номером итерации.
5. Отчитывается: что сделано, что проверено, что осталось открытым.

Definition of Done одной фичи (из `22`): есть endpoint, UI, запись в БД, данные
видны в связанных модулях, ошибки обработаны, права проверены, есть пустые
состояния, проходит ручной тест.

## 🟡 Итерация 1 — События и расписание

**Цель:** создать событие в админке и показать клиенту.

- Покрывает: `05`, `07`, `13`, `14`, `15`, `16`, `17` (Flow 6–8), `19` (Event status/правила).
- Backend: Event/Venue/Host CRUD; publish/cancel/complete с проверкой переходов
  (`17` state machine, `19` правила публикации).
- Admin: расписание (недельный вид + список), карточка события, создание/редактирование (`15`).
- Client: расписание (только published будущие) + карточка события (`16`, `03`).
- **DoD:** админ публикует событие → клиент видит его и открывает карточку.
  Проверяет: `23` тест 2 (published виден, draft скрыт).

## ⬜ Итерация 2 — Запись на событие

**Цель:** закрыть главную пользовательскую петлю.

- Покрывает: `05`, `06`, `07`, `13`, `16`, `17` (Flow 3–5), `19`, `21` (spots/quorum).
- Backend: Booking; при записи создаются Booking(`booked`) + Attendance(`unknown`) +
  Payment(`unpaid`) (`17` Flow 3); проверки записи; отмена → `cancelled` +
  attendance `cancelled_before_event`; уникальность активной записи (`14`).
- Client: кнопка «Записаться», шаг профиля при первой записи, подтверждение,
  отмена, ближайшие записи на Главной (`16`, `03`).
- Admin: список участников события (`15`).
- **DoD:** запись end-to-end; место освобождается при отмене.
  Проверяет: `23` тесты 3, 4 (duplicate), 5 (cancel), 6 (event full).

## ⬜ Итерация 3 — Профиль и пользователи

**Цель:** собрать минимальные данные и дать админу карточку пользователя.

- Покрывает: `04`, `05`, `06`, `13`, `15`, `16`, `19`, `21` (метрики пользователя).
- Backend: UserProfile (`14`); метрики пользователя (посещения, no-show, серия,
  оплачено) и статусы new/active/sleeping (`06`, `19`, `21`).
- Client: профиль + редактирование (источник не редактируется) (`16`, `06`).
- Admin: список пользователей с фильтрами/поиском + карточка пользователя со
  вкладками (`15`, `04`).
- **DoD:** пользователь меняет базовые данные; админ видит профиль/записи/метрики.

## ⬜ Итерация 4 — Посещаемость и оплаты

**Цель:** начать управлять реальными событиями.

- Покрывает: `08`, `13`, `17` (Flow 9–10), `19` (attendance/payment + финправила), `21`.
- Backend: attendance bulk; payment bulk; при `paid` → income Transaction, связанная
  с payment/user/event/cashbox (`08`, `19`).
- Admin: вкладка «Участники» — отметка явки/no-show/оплаты, сумма, касса, массовые
  действия, «применить цену события» (`15`, `08`).
- **DoD:** админ закрывает событие; в карточке пользователя видны явка и оплаты;
  событие даёт revenue. Проверяет: `23` тесты 7, 8.

## ⬜ Итерация 5 — Финансы MVP

**Цель:** видеть деньги.

- Покрывает: `10`, `13`, `14`, `15`, `17` (Flow 11), `21` (finance-метрики).
- Backend: Cashbox/Category/Transaction; ручной expense; баланс кассы = income − expense.
- Admin: экран Финансы (доход/расход/прибыль/баланс/неоплачено), таблица
  транзакций с фильтрами, добавить доход/расход/кассу; вкладка «Финансы» в событии (`15`).
- **DoD:** админ вносит расход площадки → прибыль события обновляется.
  Проверяет: `23` тест 9.

## ⬜ Итерация 6 — Управленческая Главная админки

**Цель:** рабочий операционный экран.

- Покрывает: `04`, `11`, `13`, `15`, `19` (risk status), `21` (dashboard-метрики).
- Backend: dashboard summary (upcoming, risk, expected_revenue, метрики недели).
- Admin: Главная — ближайшие события, статусы риска, блок действий, деньги/люди недели (`15`, `11`).
- **DoD:** с Главной понятно, что делать сегодня. Проверяет: `23` тесты 10, 13.

## ⬜ Итерация 7 — Пуши

**Цель:** базовая коммуникация.

- Покрывает: `09`, `13`, `14`, `17` (Flow 12), `20` (шаблоны/правила/сегменты).
- Backend: Notification + NotificationRecipients; вычисление сегментов; отправка в
  Telegram; reminders 24h/3h (ручной триггер допустим, если нет планировщика — `20`);
  защита от ошибок (`20`).
- Admin: ручная отправка по сегменту с предпросмотром получателей; история отправок (`15`).
- **DoD:** участники получают сообщение; видна история. Проверяет: `23` тест 11.

## ⬜ Итерация 8 — Опросы

**Цель:** собирать обратную связь.

- Покрывает: `09`, `13`, `16`, `17` (Flow 13), `20` (опрос MVP).
- Backend: Survey/SurveyResponse; привязка к событию; пост-ивент отправка attended.
- Client: экран опроса (rating + текст) (`16`).
- Admin: создание опроса, просмотр ответов в событии и карточке пользователя (`15`).
- **DoD:** после события собирается оценка и комментарий. Проверяет: `23` тест 12.

## ⬜ Итерация 9 — Аналитика

**Цель:** видеть рост и повторные визиты.

- Покрывает: `11`, `21` (все определения метрик).
- Backend/Admin: события по неделям, attendance/no-show, payment rate, revenue/profit;
  разрезы по активности/площадке/хосту/источнику; retention first→second→third,
  weekly active, серии недель (`21`).
- **DoD:** команда отвечает на вопросы из критерия готовности `11`.

## ⬜ Итерация 10 — CSV-экспорт и качество данных

**Цель:** переносимость и чистота данных.

- Покрывает: `10`, `11`, `13` (export endpoints), `22` (data quality), `24` (#18 удаление).
- Backend: экспорт users/events/bookings/attendance/payments/transactions/surveys;
  проверки качества; защита от дублей telegram_id; мягкое удаление статусами.
- **DoD:** ключевые данные выгружаются в читаемый CSV. Проверяет: `23` тест 14.

## ⬜ Итерация 11 — Дизайн-проход и полировка

**Цель:** довести продукт до клубного качества и операционной скорости.

- Покрывает: `25` (полное применение стиля), `16`/`15` (states), `20`/`25` (тон),
  `15` (календарь и массовые действия).
- Client: применить токены/типографику/фотостиль из `25`; все loading/empty/error
  состояния; чек экранов клиента из `25`.
- Admin: спокойный визуальный язык (не «серая CRM»), статус-бейджи, улучшенный
  недельный календарь, массовые отметки, фильтры; чек экранов админки из `25`.
- **DoD:** проходят оба чек-листа качества из `25`; regression-checks из `23`.

## Принцип приоритизации

Порядок важнее красоты: события → записи → профиль → явка/оплаты → финансы →
главная → пуши → опросы → аналитика → экспорт → дизайн (`12`). Если в итерации
успеваем мало — берём то, что сильнее закрывает текущий операционный хаос.
