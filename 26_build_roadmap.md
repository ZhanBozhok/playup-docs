# 26. Build Roadmap — исполняемые итерации

## Назначение

Этот файл превращает всю документацию (00–25) в последовательность
исполняемых итераций. Формат рассчитан на работу с AI-агентом: ты говоришь
**«сделай итерацию N»**, агент берёт её scope, строит код, сверяет результат с
Definition of Done и тест-кейсами из `23`, и коммитит.

Роадмап опирается на `12_release_plan.md` (продуктовый порядок) и
`22_development_checklist.md` (майлстоуны), но детализирован до уровня задач и
покрытия документов.

## Где живёт код (важно)

- Этот репозиторий (`playup-docs`) остаётся **только документацией** — код сюда
  не коммитим (см. `AGENTS.md`).
- Код приложений живёт в отдельных репозиториях (решение из `00`/`02`):
  - `playup-admin-app` — админка + общий backend/API (backend внутри admin-репо,
    как рекомендует `24`);
  - `playup-client-app` — Telegram mini app.
- По умолчанию агент создаёт их как соседние папки рядом с этим репозиторием и
  инициализирует git в каждой. Это решается в Итерации 0.

## Рекомендуемый стек (финализируется в Итерации 0)

- БД: PostgreSQL (`02`, `14`).
- Backend + Admin: Next.js (API routes + админ-UI в одном репо) — самый быстрый
  путь к рабочей петле; ORM Prisma. Альтернативы (NestJS/FastAPI) допустимы.
- Client: Vite + React как Telegram mini app (`03`, `22`).
- Дизайн-токены: из `25` (общий пакет/файл токенов, переиспользуется обоими UI).
- Это рекомендация. Если ты выберешь другой стек, скажи в Итерации 0 — роадмап от
  стека не зависит, меняются только инструменты.

## Как агент исполняет итерацию

1. Перечитывает документы из колонки «Покрывает».
2. Делает backend → данные → UI; не смешивает Booking/Attendance/Payment (`17`,`19`).
3. Прогоняет соответствующие manual-тесты из `23` и data-quality из `22`.
4. Коммитит в репозитории кода (не в docs), сообщение с префиксом
   `feat:`/`chore:` и номером итерации.
5. Отчитывается: что сделано, что проверено, что осталось открытым.

Definition of Done одной фичи (из `22`): есть endpoint, UI, запись в БД, данные
видны в связанных модулях, ошибки обработаны, права проверены, есть пустые
состояния, проходит ручной тест.

## Принцип приоритизации

Порядок важнее красоты: события → записи → профиль → явка/оплаты → финансы →

успеваем мало — берём то, что сильнее закрывает текущий операционный хаос.
