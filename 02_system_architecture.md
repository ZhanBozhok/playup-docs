# 02. System Architecture

## Общий подход

Система строится как два приложения поверх одной общей базы данных.

Клиентская мини-апка и админка читают и меняют одни и те же сущности: пользователей, события, записи, посещаемость, платежи, расходы, площадки, пуши, опросы и ответы.

Главный риск — две разные логики и два разных источника правды. Этого нельзя допустить.

## Репозитории

`playup-client-app` отвечает за Telegram mini app: авторизация через Telegram, ближайшие события, расписание, карточка события, запись, отмена записи, легкий онбординг, профиль, ответы на опросы.

`playup-admin-app` отвечает за внутренний интерфейс: пользователи, события, расписание, записи, явка, оплаты, площадки, расходы, кассы, пуши, опросы, дашборд, аналитика, CSV-экспорт.

## Единая база данных

Рекомендуемый тип базы — PostgreSQL.

Причины: простая реляционная модель, удобные связи между пользователями/событиями/деньгами, нормальная работа с аналитическими запросами на старте, понятный путь масштабирования.

## Backend-слой

Рекомендуемый вариант для MVP — общий API/backend-слой, даже если физически он живет в репозитории админки.

Минимальные правила:

- клиент не ходит напрямую в базу;
- админка не обходит бизнес-логику;
- критичные операции проходят через API;
- база остается единственным источником правды.

## Базовые модули

Identity — пользователи, Telegram ID, профиль, роли.

Events — события, расписание, вместимость, кворум, цена, площадка, хост, статус публикации.

Bookings — записи пользователей на события.

Attendance — факт посещения.

Payments — ручная отметка оплат.

Finance — доходы, расходы, кассы, категории.

Venues — площадки и связанные расходы.

Notifications — ручные и базовые автоматические сообщения.

Surveys — простые опросы после события.

Analytics — дашборды и метрики.

## Доступы

Клиент видит только опубликованные события, собственные записи, собственный профиль, собственную статистику и доступные опросы.

Администратор видит всех пользователей, все события, все записи, финансы, площадки, ответы на опросы и аналитику.

## Статусы события

`draft` — черновик.

`published` — видно пользователям.

`cancelled` — отменено.

`completed` — завершено.

## Не делаем в архитектуре MVP

Не нужны микросервисы, отдельное аналитическое хранилище, event streaming, сложная очередь сообщений, data warehouse, сложный RBAC, CQRS и GraphQL, если REST быстрее для команды.

## Критерий качества

Архитектура хорошая, если новый разработчик за один день понимает, где создается событие, как оно попадает пользователю, где хранится запись, где отмечается явка, где появляется доход, как строится дашборд и как выгружаются данные.
