# 06. Users, Onboarding, Profile

## Назначение

Пользовательский слой нужен для трех задач: идентифицировать человека, понять его базовые предпочтения и видеть историю участия в PlayUp.

В MVP нельзя перегружать вход длинным онбордингом. Но нельзя и совсем не собирать данные, иначе админка и аналитика будут слепыми.

## Идентификация

Основной способ идентификации — Telegram.

Система сохраняет Telegram ID, Telegram username, дату первого открытия мини-апки и дату последней активности.

Telegram username не является главным ключом. Он может измениться.

## Когда создается пользователь

User создается при первом открытии мини-апки или при первом действии, требующем идентификации.

Лучший вариант: создавать User при первом открытии, а UserProfile заполнять позже — в момент первой записи.

## Онбординг

Тяжелый онбординг до просмотра расписания не делаем.

Пользователь сначала видит ценность: события, расписание, формат клуба. Сбор данных включается в момент первой записи.

Флоу: пользователь нажимает “Записаться”. Если профиль не заполнен, система показывает короткий шаг с 3–5 полями. После заполнения запись завершается.

## Минимальные поля первой записи

Обязательные или почти обязательные:

- имя;
- уровень;
- интересующий вид активности;
- район или удобная локация;
- откуда узнал о PlayUp.

Опционально:

- телефон;
- удобные дни/время;
- комментарий.

## Уровни

На старте уровни должны быть простыми:

- новичок;
- любитель;
- уверенный;
- продвинутый.

Не нужно сразу строить сложную рейтинговую систему.

## Источник привлечения

Поле нужно для базовой маркетинговой аналитики.

Примеры: Instagram, Threads, Telegram chat, friend, host invite, ads, offline, other.

Лучше хранить как справочник или enum, но дать возможность выбрать “другое”.

## Профиль клиента

В клиентском профиле пользователь видит имя, Telegram username, уровень, интересы, район, источник привлечения, количество посещенных событий, текущую серию недель подряд и ближайшие записи.

Пользователь может менять имя, уровень, интересы, район, удобное время и телефон.

Источник привлечения лучше не делать главным редактируемым полем, чтобы не портить аналитику. Если пользователь ошибся, админ может поправить вручную.

## Карточка пользователя в админке

Админ видит User ID, Telegram ID, Telegram username, имя, телефон, уровень, интересы, район, источник привлечения, дату регистрации, последнюю активность, записи, посещения, no-show, оплаты, ответы на опросы и комментарии админа.

## Статусы пользователя

`new` — зарегистрировался, но еще не пришел.

`active` — посещал события недавно.

`sleeping` — давно не приходил.

`blocked` — заблокирован вручную.

В MVP статус можно считать простыми правилами: active — был на событии за последние 30 дней; sleeping — не был больше 30 дней; new — не имеет посещений.

## Метрики пользователя

Минимально считаем всего записей, всего посещений, всего no-show, последнюю запись, последнее посещение, количество недель подряд с посещением, суммарную оплату, средний чек и любимую активность по посещениям.

## Админские заметки

В карточке пользователя должно быть поле заметок.

Примеры: “любит футбол по воскресеньям”, “привел друга”, “часто отменяет в последний момент”, “хорошо подходит для продвинутой группы”.

## Не делаем на старте

Не делаем публичные профили, фотографии, друзей, личные сообщения между участниками, рейтинг игрока, верификацию документов, сложные анкеты, персональные рекомендации, автоматический матчмейкинг и поведенческий скоринг на основе кликов.

## Критерий готовности

Модуль готов, если пользователь создается через Telegram, может записаться после короткого заполнения профиля, админ видит всех пользователей, открывает карточку, видит профиль/записи/посещения/оплаты/no-show, а пользователь может изменить базовые данные.
