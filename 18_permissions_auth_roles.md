# 18. Permissions, Auth, Roles

## Назначение

Этот файл описывает авторизацию и права доступа. В MVP права должны быть простыми, но админка не должна быть открытой.

## Типы пользователей

В системе есть клиент PlayUp и админ PlayUp. Это разные роли с разной авторизацией.

## Клиентская авторизация

Клиент входит через Telegram mini app. Backend проверяет Telegram init data, извлекает Telegram user ID, находит или создает User и возвращает session token.

## Клиент может

- видеть опубликованные будущие события;
- видеть карточку опубликованного события;
- записаться на событие;
- отменить свою запись;
- видеть свой профиль;
- редактировать свой профиль;
- видеть свои записи;
- отвечать на свои опросы.

## Клиент не может

- видеть draft events;
- видеть чужие записи;
- видеть список всех участников;
- видеть финансы;
- видеть внутренние заметки;
- менять цену, вместимость, площадку;
- отмечать явку;
- отмечать оплату.

## Админская авторизация

В MVP можно использовать email/password или magic link. Главное — все admin endpoints защищены токеном и проверкой роли.

## Роли админов

### owner

Может всё: админы, события, пользователи, финансы, настройки, экспорт.

### admin

Операционная роль: пользователи, события, записи, явка, оплаты, расходы, площадки, пуши, опросы, аналитика.

Не может управлять owner и критичными системными настройками.

### viewer

Только просмотр. В MVP можно заложить роль, но не делать интерфейс.

## MVP-рекомендация

На старте достаточно `owner` и `admin`. `viewer` оставить в enum на будущее.

## Защита client endpoints

Каждый client endpoint определяет пользователя из токена, а не из `user_id` в query. Пользователь не может запросить чужой профиль или чужую запись.

## Защита admin endpoints

Каждый admin endpoint проверяет валидный admin token, роль и active status.

Frontend может скрывать кнопки, но backend обязан проверять права.

## Персональные данные

Храним минимум: Telegram ID, Telegram username, имя, телефон, спортивные предпочтения, уровень, район.

Не собираем паспорт, точный адрес, дату рождения без необходимости, медицинские данные и документы.

## Audit минимум

Сохранять:

- кто создал событие;
- кто отметил оплату;
- кто отметил явку;
- кто создал транзакцию;
- кто отправил пуш.

Полный audit log можно отложить.

## Не делаем на старте

Не делаем сложный RBAC, права на уровне отдельных полей, SSO, 2FA, кабинет хоста, доступ площадок и permission matrix UI.
