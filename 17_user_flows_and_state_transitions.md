# 17. User Flows and State Transitions

## Назначение

Этот файл описывает ключевые сценарии и переходы состояний. Главное — не смешивать запись, явку, оплату и движение денег.

## Flow 1. Первый вход пользователя

1. Пользователь открывает Telegram mini app.
2. Клиент отправляет Telegram init data.
3. Backend валидирует подпись.
4. Backend ищет пользователя по `telegram_id`.
5. Если пользователя нет — создает `users`.
6. Backend возвращает токен.
7. Клиент открывает Главную.

Создается только User. UserProfile можно создать пустым или создать позже при первой записи.

## Flow 2. Просмотр расписания

1. Пользователь открывает “Расписание”.
2. Клиент запрашивает published events.
3. Backend возвращает только опубликованные будущие события.
4. Draft, cancelled и completed не показываются в общем расписании.

## Flow 3. Первая запись

1. Пользователь открывает событие.
2. Нажимает “Записаться”.
3. Если профиль не заполнен, клиент показывает короткую форму.
4. Пользователь заполняет имя, уровень, спорт, район, источник.
5. Backend обновляет UserProfile.
6. Backend проверяет событие и места.
7. Backend создает Booking `booked`.
8. Backend создает Attendance `unknown`.
9. Backend создает Payment `unpaid`.
10. Клиент показывает подтверждение.

Так в админке участник сразу отображается с явкой и оплатой.

## Flow 4. Повторная запись

1. Пользователь нажимает “Записаться”.
2. Профиль уже заполнен.
3. Backend проверяет событие.
4. Создает Booking, Attendance, Payment.
5. Клиент показывает подтверждение.

## Flow 5. Отмена записи пользователем

1. Пользователь нажимает “Отменить запись”.
2. Подтверждает.
3. Backend меняет Booking на `cancelled`.
4. Backend ставит `cancelled_at`.
5. Attendance меняется на `cancelled_before_event`.
6. Место освобождается.

Если Payment уже `paid`, возврат не делается автоматически в MVP. Это ручное действие админа.

## Flow 6. Создание события админом

1. Админ открывает Расписание.
2. Кликает по слоту или “Создать событие”.
3. Заполняет поля.
4. Сохраняет draft.
5. Публикует, если обязательные поля заполнены.
6. Событие появляется у клиента.

Переход: `draft -> published`.

## Flow 7. Отмена события админом

1. Админ открывает событие.
2. Нажимает “Отменить”.
3. Указывает причину.
4. Система меняет status на `cancelled`.
5. Система предлагает отправить сообщение участникам.
6. Клиент видит отмену, если был записан.

Переходы: `draft -> cancelled`, `published -> cancelled`.

## Flow 8. Закрытие события

1. Событие прошло.
2. Админ открывает карточку.
3. Отмечает явку.
4. Отмечает оплаты.
5. Добавляет расходы.
6. Отправляет опрос.
7. Нажимает “Завершить”.
8. Status становится `completed`.

Переход: `published -> completed`.

## Flow 9. Отметка явки

1. Админ открывает вкладку “Участники”.
2. Ставит `attended`, `no_show` или `cancelled_before_event`.
3. Backend обновляет Attendance.
4. Карточка пользователя и аналитика используют новый статус.

## Flow 10. Отметка оплаты

1. Админ ставит Payment `paid`.
2. Указывает сумму и кассу.
3. Backend обновляет Payment.
4. Backend создает Transaction `income`.
5. Transaction связывается с Payment, User, Event и Cashbox.

## Flow 11. Добавление расхода

1. Админ открывает Финансы или карточку события.
2. Нажимает “Добавить расход”.
3. Указывает сумму, кассу, категорию, дату.
4. Опционально указывает событие, площадку, хоста.
5. Backend создает Transaction `expense`.

## Flow 12. Ручное сообщение

1. Админ выбирает сегмент.
2. Пишет текст.
3. Backend определяет получателей.
4. Создает Notification.
5. Создает NotificationRecipients.
6. Отправляет сообщения.
7. Сохраняет статусы.

## Flow 13. Опрос

1. Админ создает опрос.
2. Отправляет участникам.
3. Пользователь отвечает.
4. Backend создает SurveyResponse.
5. Ответ виден в событии и карточке пользователя.

## State machine: Event

Разрешено:

- `draft -> published`
- `draft -> cancelled`
- `published -> cancelled`
- `published -> completed`
- `completed -> published` только как ручная коррекция
- `cancelled -> draft` только как ручная коррекция

Запрещено по умолчанию:

- `completed -> cancelled`
- `cancelled -> completed`

## State machine: Booking

Разрешено:

- `booked -> cancelled`
- `booked -> admin_removed`
- `waitlisted -> booked` позже
- `cancelled -> booked` только через новую запись или ручное восстановление.

## State machine: Attendance

Разрешено:

- `unknown -> attended`
- `unknown -> no_show`
- `unknown -> cancelled_before_event`

Админ может исправить статус вручную.

## State machine: Payment

Разрешено:

- `unpaid -> paid`
- `paid -> refunded`
- `unpaid -> waived`
- `paid -> waived` только ручная коррекция
- `refunded -> paid` только ручная коррекция

## Инварианты

Booking не означает attendance. Attendance не означает payment. Payment не означает attendance. Published event должен иметь публичные поля. Completed event должен иметь отмеченную явку, но в MVP это warning, а не hard block.
